The foundation of the solution that enables you to tackle these business challenges is an emerging standard from the Object Management Group (OMG) called Model-Driven Architecture (MDA). MDA’s vision is to define a new way of specifying and building systems which is based on modeling with UML. MDA captures the use of UML for more than just pretty pictures. For example, in the past and even still today designers model a system and these pretty pictures are collected and bound into a document and sent to the developers as a documentation set. Developers pick out what they think is valuable and leave the rest. In addition, you see that over time the paper model is not maintained when the application changes, so model and code are no longer in sync. The model becomes useless as a documentation set because it no longer reflects the application accurately. MDA’s vision is to merge the modeling and coding worlds.
OMG MDA changes the way organizations design and develop software by separating an application’s  business logic from the infrastructure on which it runs. •Based on modeling with UML: Traditional software design and development focuses  on creating applications that are deployed on a specific platform. MDA enables organizations to model (in UML) at a higher level of abstraction. In essence the models are the application “blue-prints”, independent of any particular hardware, operating system or middleware. •Modeling instead of programming: This approach ensures that business models drive application development, instead of the other way around. The ultimate goal: To truly bridge modeling and development.  •Merging modeling and coding: Modeling, however, is still a paper-based exercise for a lot of organizations. The result is that, although the model is intended to be the application blueprint, as the application evolves the model and code are no longer synchronized because the application is updated and the model isn’t. Therefore, from a documentation perspective the model becomes useless. This is what MDA tries to address: Merging the modeling and coding worlds. •Based on standards: MDA makes use of a strong set of universal standards for modeling that makes it a very useful and interesting concept. MDA is built on UML, the Meta Object Facility (MOF), XML Metadata Interchange (XMI), and the Common Warehouse Metamodel (CWM)
MDA is not the next big thing. Let me explain.
The problem is that every year we see a new infrastructure being introduced. If we look at the last 10 years; C/C++, Java, J2EE, Web Services, .NET, and who knows what’s coming out next year.  The nice thing about MDA is that it clearly separates the business model from the architecture model and it separates the architecture model from the implementation language. This makes it very clear to determine where to apply changes in the application:
1.Business changes on the business model
2.Architecture changes on the architecture model
3.Code changes on code model
Addition information for the speaker:
Dr Richard Soley is Chairman and CEO of the Object Management Group (OMG)
MDA addresses the challenges of today's highly networked, constantly changing systems environment, providing an architecture that assures: •Portability, increasing application re-use and reducing the cost and complexity of application development and management, now and into the future. •Cross-platform Interoperability, using rigorous methods to guarantee that standards based on multiple implementation technologies all implement identical business functions. •Platform Independence, greatly reducing the time, cost and complexity associated with re-targeting applications for different platforms-including those yet to be introduced. •Domain Specificity, through Domain-specific models that enable rapid implementation of new, industry-specific applications over diverse platforms. •Productivity, by allowing developers, designers and system administrators to use languages and concepts they are comfortable with, while allowing seamless communication and integration across the teams.
The benefits of MDA are significant-to business leaders and developers alike:
•Reduced cost throughout the application life-cycle
•Reduced development time for new applications
•Improved application quality
•Increased return on technology investments
•Rapid inclusion of emerging technology benefits into their existing systems

MDA provides a solid framework that frees system infrastructures to evolve in response to a never-ending parade of platforms, while preserving and leveraging existing technology investments. It enables system integration strategies that are better, faster and cheaper.
 •Crucial return on investment: applications and domain models can be reused and adopted to ongoing business and technology changes.  It allows a software environment to evolve and change while the business model (PIM) remains stable. This stable base is necessary to cope with the constantly changing environment of today’s market.

At early phase the compiler doesn’t do a good transformation from high-level language into assembly language. However, as technology evolved, we know that the compiler can do as good as we did before and the systems nowadays are too complex to do it manually.
The same paradigm can be applied to MDA. When the transformation definition & the tool evolved, we no longer need to afraid the quality of code generated.
